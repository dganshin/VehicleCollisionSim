# 车辆碰撞虚拟仿真系统五人分工方案

> 本分工按当前仓库实际代码、`README.md`、`docs/` 开发记录和测试记录整理。  
> 当前项目不是空白模板，已经包含车辆控制、多车切换、车车碰撞、环境碰撞、测试树、运行时 UI、相机、白天/黑夜、README、测试用例和 AI 使用记录等内容。  
> 队长负责最高贡献和核心技术主线，其它成员围绕真实存在的模块做专项整理、验证、补充截图、PPT 小节和结题报告小节。

## 一、项目当前实际功能概览

根据当前仓库内容，本项目已经实现或记录了以下功能：

- Unity 车辆碰撞虚拟仿真系统，最终场景为 `Assets/Scenes/CollisionTestScene.unity`
- 当前 README 中记录了 4 辆车：`Car1`、`Car2`、`Car3`、`Car4`
- 支持键盘驾驶：
  - `W` 前进
  - `S` 后退
  - `A/D` 转向
  - `Space` 刹车
  - `1/2/3/4` 切换车辆
  - `R` 重置当前车辆
  - `Shift + R` 重置全部车辆
  - `F2` 显示或隐藏参数面板
- 支持车辆与车辆的物理碰撞
- 支持车辆与环境对象的物理碰撞：
  - 城市道路
  - 马路牙子
  - 人行道柱
  - 桥梁
  - 建筑边界
  - 障碍物
  - 测试树 `CrashTestTree`
- 支持 3 棵可碰撞测试树，用于植物碰撞演示
- 支持运行时 UI 参数面板：
  - 当前车辆名称
  - 速度 `m/s` 和 `km/h`
  - 前向速度
  - 估算前向加速度
  - 估算纵向合力
  - 质量
  - 重力加速度
  - 估算车长
  - 翻车状态
  - 车辆切换按钮
  - 重置按钮
  - 白天/黑夜切换
  - 驱动力、刹车、阻尼、摩擦、碰撞推车等参数滑条
  - 相机参数调节
- 支持第三人称相机跟随和鼠标绕车观察
- 支持视觉轮子旋转
- 支持运行时或 Editor 辅助安装车辆、测试树和最终场景
- `docs/` 中已经记录大量 AI / Code Agent 使用过程、Bug 修复过程和测试用例

## 二、PPT 和结题报告的 N+1 结构

建议 PPT 和结题报告采用 `N+1` 结构：

- `+1`：项目整体结构，由队长统一负责和最终汇总。
- `N=5`：五名成员各自负责一个模块小节。

每个人负责自己模块的 PPT 页面和结题报告小节，至少包含：

- 模块目标
- 负责范围
- 对应代码文件或资源文件
- 实现方法
- 测试方法
- 截图或录屏材料
- 使用 AI / Code Agent 的过程说明
- 遇到的问题和解决办法

## 四、队长：系统总体设计、车辆物理控制核心与最终集成

### 负责定位

队长负责本项目最高贡献部分。核心车辆控制、碰撞主线、AI Agent 调试链路、Git 管理、最终演示效果和总集成都归队长负责。

### 主要负责内容

- 负责项目总体技术路线：
  - 从课程要求拆解出车辆控制、车车碰撞、环境碰撞、UI 参数面板、文档交付等模块
  - 最终确定使用 Unity、PhysX、Rigidbody、WheelCollider、C# 实现
  - 记录 PyBullet、UE5、Unity 等路线比较和最终选择原因
- 负责车辆控制核心：
  - `W/S/A/D/Space` 输入控制
  - 新 Input System 接入
  - 车辆前进、倒车、转向、刹车
  - 起步辅助
  - 高速转向衰减
  - 低速换向响应
  - 抵墙、顶车后 W/S 反向迟滞修复
  - 前进后立刻倒车时反向驱动增强
- 负责 WheelCollider / Rigidbody 车辆主线：
  - 从早期简化刚体驱动切换到优先使用资源自带 `WheelCollider`
  - 保留无轮子时的备用刚体驱动逻辑
  - 处理轮子滚阻、轮胎摩擦、驱动轮、转向轮和制动力
- 负责车车碰撞核心：
  - 低速追尾
  - 静止目标车启动辅助
  - 贴车持续推车
  - 纵向冲量转速度
  - 非激活车辆被撞时的短时低阻力 / 低抓地窗口
  - 中低速车车碰撞可见位移
  - 高速碰撞后的姿态变化
- 负责关键稳定性问题修复：
  - 黄车埋地
  - 车辆出生高度校正
  - 切车 `count=1` 问题
  - `NullReferenceException` 初始化顺序
  - 多车参数不同步
  - 空挡滑行突然刹停
  - 未激活车辆像手刹锁死
  - UI 参数范围和实际代码参数不一致
  - Unity Crash Reporter / PhysX 材质重复重挂崩溃
  - 代码改了但场景实例仍吃旧 Inspector 序列化值
- 负责 Git 管理：
  - 保存项目快照
  - 提交关键版本
  - 保证修改可回滚
- 负责 AI / Code Agent 协作主线：
  - 向 Agent 描述问题
  - 让 Agent 搜索代码和场景文件
  - 让 Agent 修改脚本
  - 人工进入 Unity 测试
  - 继续根据测试反馈调整
  - 将过程写入 PPT 和结题报告

## 十二、注意事项

- 不要写成所有人都平均完成核心代码。核心车辆控制和系统集成应明确由队长主导。
- 其它成员可以写“负责整理、验证、测试、配置、演示材料和报告小节”，这样更符合真实协作和最终答辩需要。
- PPT 和结题报告必须写清楚 AI / Code Agent 的使用过程，这是课程要求中的关键点。
- 不要承诺工业级真实车损、真实发动机、差速器、完整悬挂系统；当前项目定位是课程级车辆碰撞虚拟仿真。
- 最终演示优先保证：
  - 4 辆车都能驱动和切换
  - 车车碰撞有可见物理反馈
  - 车辆能撞环境对象和测试树
  - UI 能展示和调节关键物理参数
  - README 能让老师快速复现
