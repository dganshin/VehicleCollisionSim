# 开发时间线

说明：本时间线用于后续撰写 PPT、结题报告和项目复盘。内容以“关键节点 + 决策原因”为主，强调 AI 为辅助工具，核心路线由人工选择和整合。

## 关键节点

1. 前期确定课程项目方向为“基于 Unity 的车辆碰撞虚拟仿真与物理参数可视化系统”。
2. 尝试使用 PyBullet 做原型验证，用于快速测试基础车辆物理与碰撞思路。
3. 在原型体验中发现 PyBullet 更偏研究和验证，视觉表现、驾驶感与课程展示效果不足。
4. 转而评估 UE5，希望获得更强的画面和交互表现。
5. 在 UE5 评估过程中发现工具链、工程结构与学习成本较高，不适合短期课程交付。
6. 综合课程周期、可控性、资源生态和组员实现成本，最终选择 Unity 作为正式开发平台。
7. 创建 Unity 工程并确定以车辆控制、碰撞仿真、UI 参数调节、物理可视化为主要模块。
8. 导入免费车辆资源 `ARCADE - FREE Racing Car`，用于快速搭建可展示场景。
9. 遇到导入资源材质显示异常，采用当前渲染管线下的临时兼容方案，先解决粉色材质问题，保证模型可展示。
10. 在车辆控制接入过程中，发现 `Input.GetAxis` 与 Unity 新 Input System 冲突。
11. 调整技术路线，决定优先基于新 Input System 实现键盘控制，而不是回退旧输入方案。
12. 编写并接入 `SimpleCarController`，支持 `W/S/A/D/Space` 的基础车辆操作。
13. 排查车辆对象层级、`Rigidbody`、碰撞体与脚本挂载关系，明确控制脚本应放在 `bigyellowbee` 根对象。
14. 为解决 Game 视角固定问题，补充 `CameraFollow` 跟随脚本方案。
15. 发现默认 `AddForce` 在车体质量较大时运动不明显，影响课程演示效果。
16. 将车辆运动逻辑改为 `ForceMode.Acceleration`，并加入限速思路，使车辆在当前参数下能够更明显地移动和碰撞。
17. 开始同步建立 `docs/` 文档记录体系，用于沉淀 AI 使用记录、测试用例、模块分工与 README 要点。
18. 在进一步测试中发现刚体车方案仍存在低速卡地、高速相机抖动、轮子不转、满舵易翻和缺少复位/切车等问题。
19. 结合场景和 prefab 检查，确认车辆当前方案里保留的 `WheelCollider` 不是主控，但其激活状态可能干扰当前简化刚体车表现。
20. 继续坚持课程级“简化刚体车辆模型”路线，不切换到复杂 WheelCollider 主线，而是在现有 `Rigidbody + Collider + 简化控制脚本` 基础上做稳定化修正。
21. 在 `SimpleCarController` 中加入起步辅助、平面方向驱动、低重心、侧向速度阻尼、高速转向衰减和翻车判定。
22. 重写 `CameraFollow` 为平滑第三人称跟随，只跟随位置与 yaw，不继承车辆 roll/pitch。
23. 新增 `VehicleManager`，实现车辆切换、当前车辆控制权切换、相机目标更新和 `R` / `Shift+R` 复位。
24. 新增 `WheelVisualRotator`，在不改动物理主线的前提下补充轮子视觉旋转，改善“船/火箭艇”观感。
25. 用户向场景中新增第二辆红车后，发现它尚未接入控制脚本和车辆管理器，因此无法切换和控制。
26. 继续沿用简化刚体方案，为 `Car2` 补充控制脚本和视觉轮子脚本，并把出生高度从接近地面改为更安全的初值。
27. 为降低“方块滑动摩擦”感，在控制脚本中为车辆碰撞体加入运行时低摩擦材质，并让车辆管理器自动发现并合并场景中的车辆。
28. 在多车测试中发现自动发现逻辑会按名字重排车辆顺序，因此进一步修正 `VehicleManager`，保留 Inspector 中的人为顺序，并补充小键盘数字切车支持。
29. 在用户实际体验中发现上一版更像“船式转向”，因此继续回调控制模型：限制低速转向、增加 yaw 阻尼、降低默认加速度和最高速度，使车辆更接近“先滚起来再转向”的行为。
30. 在进一步排查“黄车嵌地、黑车正常”的差异时，确认两车场景实例的 `Rigidbody.mass` 实际并不一致；随后把黄车质量从 `1200` 调回与黑车一致的 `1000`，先统一悬挂压缩基线。
31. 在继续排查“按 `1/2` 无法切车”时，直接核对 `CollisionTestScene.unity`，发现 `VehicleManager.vehicles` 场景列表实际上只保存了黄车一项，说明问题不在按键本身，而在场景接入状态不完整。
32. 同一轮检查还发现黄车场景实例仍保留旧控制参数和 `disableChildWheelColliders=1`，而 `Car2` 没有场景级控制脚本与视觉轮子脚本挂载，导致两车表现持续不一致。
33. 为避免后续每加一辆车都手动漏配，进一步修改 `VehicleManager`：运行时自动识别带 `Rigidbody + Wheels/WheelCollider` 的车辆根对象，并补齐 `SimpleCarController` 与 `WheelVisualRotator`。
34. 同步把黄车的场景参数回调到当前脚本默认值，并把 `Car2` 的初始 `Y` 高度重新抬到统一基线，减少“黑车正常、黄车嵌地”的场景级差异。
35. 用户继续反馈黄车在 Play 后仍明显埋地，并提供了 `Switch request from digit2: index=1, count=1, target=null` 的真实日志，说明当前问题已经不是输入焦点，而是车辆列表仍只有 1 辆。
36. 基于该日志进一步调整方案：不再只依赖 `FindObjectsByType<SimpleCarController>` 发现车辆，而是改为按带 `Rigidbody` 的车辆根对象收集控制器，保证像 `Car2` 这样的后加车辆也能稳定进入列表。
37. 同时针对“车轮肉眼已经被地面吃掉”的现象，在 `SimpleCarController` 中加入启动离地校正：按轮子最低接地点或碰撞体最低点自动把整车抬到地面上方，避免一进 Play 就先与地面穿插。
38. 在离地和切车问题基本定位后，用户继续反馈“车辆切换正常，但现在卡死不动，只剩夸张前后倾”，说明新的主要问题已经从接入状态转向驱动模型本身。
39. 为避免继续使用“轮子支撑 + 车身硬推”的混合模式，进一步把 `SimpleCarController` 改成最简 `WheelCollider` 驱动：检测到现成轮子碰撞器后，直接给驱动轮 `motorTorque`、给前轮 `steerAngle`、给全轮 `brakeTorque`，只在没有 `WheelCollider` 的情况下才退回到刚体 `AddForce/AddTorque` 备用方案。
40. 为了便于课程答辩时解释“每一步是怎么利用 AI 辅助完成的”，继续在核心脚本中补充说明性注释，把参数用途、控制逻辑分支、车辆发现方式和相机设计意图直接写回代码。
41. 在进一步体验测试中，发现低速松油门会出现“突然像被踩一脚刹车”的感觉，因此回查到 `ApplyWheelDrive` 里的自动补刹车逻辑，并将其改为可选开关，默认关闭。
42. 用户又提出跟车视角不能像 GTA 一样看车辆侧面，于是继续扩展 `CameraFollow`：保留第三人称跟车基础上，加入鼠标绕车旋转的 yaw/pitch 控制和俯仰限制。
43. 在碰撞展示测试中，发现未激活车辆即使顺着轮子方向被追尾，也显得过于“粘地”，像始终拖着手刹，因此继续把激活车与非激活车的滚动阻力拆开处理。
44. 具体实现上，在 `SimpleCarController` 中新增 `inactiveLinearDamping` 和 `inactiveWheelDampingRate`，让当前控制车辆的滑行手感与其他车辆的被撞动难易程度可以分别调节。
45. 在继续测试多车输入时，又发现未激活车辆偶尔会像当前车辆一样响应输入，因此继续从控制权分配机制入手，而不是只调物理参数。
46. 为此把 `SimpleCarController.isControlled` 默认值改为 `false`，并在 `VehicleManager.Update()` 中每帧执行一次控制权收口，确保只有当前车辆会持续保留可控状态。
47. 在随后的实测中又出现了 `NullReferenceException`，定位到 `VehicleManager` 在 `Awake()` 阶段强制切控制权时，部分运行时补挂的 `SimpleCarController` 还没完成自身引用初始化。
48. 为消除这类初始化顺序问题，继续把 `SimpleCarController` 改成懒初始化安全版，在被 `VehicleManager` 提前调用时也能自行补齐 `Rigidbody` 和轮子缓存，不再因为空引用中断切车流程。
49. 在继续测试碰撞展示时，发现只降低未激活车辆的线性阻尼和轮子滚动阻力仍不足以让其被顶动，说明真正把车辆“钉在地上”的更大因素是 `WheelCollider` 自带的轮胎抓地力。
50. 因此继续在 `SimpleCarController` 中把未激活车辆的轮胎前向/侧向摩擦曲线也拆成单独参数，通过降低摩擦曲线 stiffness，让未激活车辆在展示碰撞时更容易被推动。
51. 在用户把代码默认值调到 `motorAcceleration=1800`、`startBoostAcceleration=2400` 后，体感仍几乎没变化，于是进一步核对场景文件，确认黄车场景实例仍保留旧的 Inspector 序列化值，而不是吃脚本新默认值。
52. 为消除这种“源码默认值已改、场景实例仍跑旧值”的错位，继续直接修改 `CollisionTestScene` 中黄车实例的序列化参数，并把未激活车辆的抓地力与滚动阻力进一步下调，统一当前测试场景的基线。
53. 在用户进一步区分“滑行正常但加速发软”与“刹停后反向会先顺着旧方向滑一点”的问题后，继续把注意力从单纯改动力参数转向阻尼策略和换向逻辑。
54. 具体实现上，在 `SimpleCarController` 中新增 `drivingLinearDamping`，让有油门时使用更低阻尼，同时加入换向阶段专用的速度阈值与刹车扭矩，使车辆先刹停再反向驱动。
55. 在随后测试中，用户发现换向蹭动反而更明显，继续回查后确认是轮驱动模式把 `motorAcceleration/startBoostAcceleration` 又额外乘了 80 倍，导致在把 Inspector 参数提得很高之后，实际轮驱动扭矩完全失真。
56. 因此继续收回这层额外放大，把 WheelCollider 驱动扭矩改为直接使用参数值，并将换向阈值和换向刹车扭矩同步调回更稳妥的一档。
57. 为了满足课程项目“UI 界面交互性友好”的要求，开始补充运行时参数面板，而不是只依赖 Inspector 调参。
58. 新增 `VehicleRuntimeUI`，直接在 Game 视图中显示当前车辆速度、估算前向加速度、估算驱动力、质量、重力加速度和估算车长，并统一用 `m`、`m/s`、`km/h`、`kg`、`N` 等单位表达。
59. 为了让 UI 能真正实时改参，继续在 `SimpleCarController` 中补充速度读数接口和运行时配置刷新入口，在 `VehicleManager` 中自动挂接 UI 面板。
60. 在 UI 交互层面继续补上当前车辆复位、全部车辆复位和车辆切换按钮，使课程演示时可以直接在 Game 视图中完成调参与切换，无需频繁切回 Inspector。
61. 在进一步试驾中，用户发现前进加速偏慢、倒车反而略快、刹车过猛且两车手感不一致，于是继续把问题从“单一动力参数不足”收敛为“前后驱动力未分离 + 场景车和运行时接入车参数来源不同”。
62. 为解决这个问题，继续在 `SimpleCarController` 中拆出倒车驱动力倍率和刹车扭矩倍率，并在 `VehicleManager` 中让运行时新接入车辆自动复制模板车的调参结果和刚体设置，尽量把场景主车与后加车辆拉回同一条参数基线。
63. 在碰撞主线测试中，用户进一步指出“运动中的车可以被撞动，但完全静止的车像焊死在地上”，于是继续把问题从一般阻力调参收敛到“静止到运动的转换”和“轮子空转换向”的处理。
64. 为此继续在 `SimpleCarController` 中加入非激活静止车的碰撞启动辅助，以及“车身没动但轮子仍按旧方向空转”时的轮速刹停逻辑，目标是让静止目标车也能被低速推动，同时让顶住另一辆车后的前后换向更像真实车辆而不是像船一样慢吞吞。
65. 在进一步实测中，发现仅靠“目标静止车被撞后自己起步”的辅助仍不够稳定，于是继续给当前受控车辆加入低速贴住目标时的直接推车辅助，让碰撞主线优先满足“低速顶车也能把静止车推出去”的演示需求。
66. 同一轮中继续把“低速受阻时轮子还按旧方向空转”的问题进一步收紧，新增短时间强制刹轮锁定，避免顶住目标后前后切换依旧发呆。
67. 在进一步定位“静止车低速撞不动”的根因后，人工明确要求碰撞主线回到“沿车辆纵向冲量传递”的解释方式，而不是继续通过把被撞车长期调成低摩擦状态来制造位移。
68. 在继续修改碰撞主线前，先把当前 Unity 项目整理成可协作的 Git 快照：补齐 `.gitignore`，提交本地仓库，为后续 GitHub 协作和问题回滚保留基线。
69. 随后重构低速碰撞逻辑：取消默认依赖“非主控车临时低阻力窗口”，改为由当前主控车在前后保险杠区域根据纵向冲量和静止贴车步进直接推动目标车，更贴近课程中“车辆与车辆物理碰撞”的汇报口径。
